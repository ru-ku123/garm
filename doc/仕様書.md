# パスワード管理ツール 仕様書

ローカル環境で動作するパスワード管理ツールの仕様書。

## 0. 前提とスコープ
- 本ツールは「単一ユーザー」「ローカルでの安全な保管」を主目的とする。
- ネットワーク接続なしで利用できる（オフライン・ファースト）。
- クラウド連携は将来の拡張（MVPでは必須にしない）。

### 0.1 MVPの方針
- MVPはGUIアプリとして提供する。
- 「自分が良いと思ったGUIのパスワード管理ツール」を目標に、見た目/操作感を重視する（ただしセキュリティ要件は必ず満たす）。

### 0.2 非スコープ（当面やらない）
- 複数ユーザーの共有ボルト、権限管理、チーム機能
- ブラウザ拡張による自動入力（将来検討）
- サーバーを用意しての同期（将来検討）

## 1. 概要
ユーザーがパスワード等の秘密情報を安全に保存・検索・生成できるツールを提供する。
保存データは暗号化された「ボルト（Vault）ファイル」としてローカルに保持し、マスターパスワードでのみ解錠できる。

## 2. 用語
- Vault: 暗号化されたデータ保管ファイル
- マスターパスワード: Vault解錠に用いるパスワード
- KDF: 鍵導出関数（パスワードから暗号鍵を作る）
- AEAD: 認証付き暗号（改ざん検知を含む暗号）
- TOTP: 時間ベースのワンタイムパスワード

## 3. 機能要件

### 3.1 Vault作成・解錠・ロック
- ユーザーはVaultを新規作成できる。
- ユーザーはマスターパスワードでVaultを解錠できる。
- 一定時間操作がない場合、自動ロックできる（デフォルト例：5分、設定可能）。

### 3.2 エントリ（アカウント情報）保存
- ユーザーは複数のエントリを保存できる。
- エントリ項目: サービス名、ユーザー名、パスワード、メールアドレス、URL、メモ、タグ（複数）。
- エントリはフォルダ階層（パス）に所属できる（例: `Work/GitHub`）。
- フォルダの作成/リネーム/削除、およびエントリの移動（フォルダ間移動）をサポートする。
- エントリの追加/更新/削除/閲覧ができる。

### 3.3 検索・フィルタリング
- サービス名、ユーザー名、タグで検索・フィルタリングできる。
- 大文字小文字の扱い、部分一致/前方一致などの検索仕様を定義し、UI上でユーザーに分かるようにする。

### 3.4 パスワード生成
- ユーザーはカスタマイズ可能なパスワードを生成できる。
- 生成オプション: 長さ、使用文字種（英大/小、数字、記号）、除外文字、最小文字種数。
- 乱数はOSのCSPRNGに基づく。

### 3.5 インポート・エクスポート
- CSV形式でのインポートをサポートする。
- CSVエクスポートは「平文での出力」であることを明示し、ユーザーの明確な操作（確認）を要求する。
- Vault自体のエクスポートは「暗号化されたVaultファイルのコピー」として扱う。

### 3.6 バックアップ・リストア
- ユーザーはVaultファイルをバックアップ（コピー）できる。
- バックアップからのリストア（復元）手順を提供する。

### 3.7 クラウド同期（将来拡張・オプション）
- MVPでは「Google Drive等のAPI連携」は行わない。
- 代替案として、ユーザーがVaultファイルを任意の同期フォルダに置く運用は妨げない（ただし競合・同時編集は未対応）。

### 3.8 多要素認証（将来拡張・オプション）
- Vault解錠時の追加要素としてTOTPを用いる構成を検討可能にする。
- TOTPの秘密鍵もVault内に保存され、解錠後にのみ参照できる。

### 3.9 GUI要件（MVP）

#### 3.9.1 画面構成
- 初回/起動画面: Vault作成 or 既存Vaultを開く
- 解錠画面: マスターパスワード入力、解錠失敗の明確な表示
- メイン画面: 3ペイン（左: ツリー、中央: 詳細、右: タグ/絞り込み）
- エントリ詳細/編集画面: 項目編集、保存/キャンセル
- パスワード生成ダイアログ: オプション指定→生成→適用
- 設定画面: 自動ロック時間、表示設定など
- インポート/エクスポート画面（またはダイアログ）

#### 3.9.2 機微情報の表示とコピー
- パスワードはデフォルトでマスク表示とする。
- 「表示/非表示」切り替え（もしくは押している間だけ表示）を提供する。
- 「コピー」操作を提供する場合、コピー後に自動でクリップボードを消去する。
  - デフォルト例: 30秒（設定可能）

#### 3.9.3 ロック操作
- メイン画面から手動ロックできる。
- 自動ロックまでの残り時間/状態がユーザーに分かることが望ましい。

#### 3.9.4 破壊的操作の確認
- エントリ削除、CSVエクスポート（平文出力）は確認ダイアログを必須とする。

#### 3.9.5 レイアウト（3ペイン）
- 左ペイン: フォルダ階層（パス）＋エントリのツリーを表示する。
- 中央ペイン: 選択中アイテム（フォルダ or エントリ）の詳細情報を表示する。
  - エントリ選択時は、項目表示と編集導線（編集/保存/キャンセル）を提供する。
  - フォルダ選択時は、フォルダ情報とフォルダ配下の概要（件数など）を表示してよい。
- 右ペイン: タグ一覧を表示し、絞り込みに利用する。右ペインは非表示にできる。

#### 3.9.6 アイテム選択ルール
- 基本的にアイテム選択は左ペインのツリーからのみ行う。
  - 検索/タグ絞り込みは「ツリー表示に反映」する方式とし、中央ペインに別の一覧を出して選択させない。
- 中央ペインは「表示/編集の場」であり、選択状態の起点にならない。

#### 3.9.7 検索・タグ絞り込みの表示ルール
- 検索とタグ絞り込みは併用できる。
- タグ絞り込みは、複数選択時デフォルト"OR"で動作し、"AND"に切り替え可能とする。
- 絞り込み結果は左ペインのツリーに反映する。
  - 条件に合わないエントリはツリー上で非表示（除外）とする。
  - フォルダは、配下に表示対象が存在する場合に限り表示する（空フォルダは検索中のみ非表示）。
- ユーザーが現在の絞り込み状態を認識できるよう、検索語と選択中タグをUI上に表示する。

#### 3.9.8 ドラッグ&ドロップ（左ペイン）
- 左ペインのツリー上で、ドラッグ&ドロップによりエントリ/フォルダを移動できる。
- 移動は「所属フォルダ（パス）の変更」として扱う。
- 不正な移動（例: フォルダを自分の子に移動、循環参照）は防止する。

#### 3.9.9 新規作成・リネーム・コンテキストメニュー
- 新規作成は左ペインで「親フォルダ（追加先）」を選択してから行う。
- 左ペインの右クリック（コンテキストメニュー）から、以下の操作を可能にする。
  - 新規エントリ作成
  - 新規フォルダ作成
  - リネーム（エントリ/フォルダ）
  - 削除（エントリ/フォルダ）

## 4. 非機能要件

### 4.1 セキュリティ
- Vaultは保存時に必ず暗号化され、改ざん検知ができること（AEAD）。
- 復号はマスターパスワードから導出した鍵でのみ可能とする。
- マスターパスワードは「ハッシュだけ保存して終わり」にしない。
  - 解錠はKDFで導出した鍵＋AEADの検証で判定する（もしくは検証用のMACを併用する）。
- 推測攻撃を遅延させるため、KDFは計算/メモリコストを持つ方式を用いる。
- 復号後の秘密情報は可能な範囲でメモリ上から消去する（例：ゼロ化）。
- クリップボードへのコピー機能を提供する場合、一定時間で自動消去する（将来対応でもよい）。
#### 4.1.1 マスターパスワード要件（ガイドライン）
- 最低8文字以上を推奨する。
- ローカル環境用ツールであるため、非常に強力なパスワードを必須にしない（ユーザビリティ重視）。以下のガイドラインによるチェックは行うが、厳格に強制しない。
	- 最低12文字以上を推奨する。
	- 文字種の多様性を推奨する（英大/小、数字、記号など）。
	- 辞書攻撃を防ぐため、一般的なパスワードは避けるよう促す。

### 4.2 パフォーマンス
- 数千件規模のエントリでも、検索・一覧表示が体感的に遅くないこと。

### 4.3 ユーザビリティ
- 直感的で使いやすいUIを提供する。
- 初回起動時に「Vault作成」導線が分かりやすいこと。

### 4.4 クロスプラットフォーム
- Windows / macOS / Linux での動作を目標とする。
- Android / iOS は将来の拡張として扱う。

### 4.5 可用性
- オフラインで完結して動作すること。
- Vault破損時に備え、バックアップ運用を案内できること。

## 5. データ形式・暗号設計（実装に必要な最低限）

### 5.1 Vaultファイル
- 単一ファイルとして保存する。
- 平文JSON/CSVをそのまま保存しない。内部構造はJSON等でもよいが、ファイル上は暗号文として格納する。

### 5.2 暗号方式
- 暗号はAEADを用いる。
  - 例: AES-256-GCM（鍵256bit、nonce 96bit）
- nonceは暗号化ごとに一意で、OSのCSPRNGから生成する。

### 5.3 鍵導出（KDF）
- マスターパスワードからVault暗号鍵を導出する。
- 推奨: Argon2id（パラメータ(OWASP推奨値を参考)は実測で調整し、Vaultヘッダに保存する）。
- saltはランダムに生成し、Vaultヘッダに保存する。

### 5.4 互換性・バージョニング
- Vaultファイルにフォーマットバージョンを持たせ、将来の移行に備える。

## 6. 技術要件
- 開発言語: Rust
- UI: GUI（MVP）
  - 候補: egui / Tauri / iced / slint
  - MVPでは実装スピード重視で egui を第一候補とする（変更可能）。
    - ただし、UIの実装検証（D&D等）で困難があればTauriへの移行も視野に入れる。
- 主要クレート例（候補）
  - AEAD: aes-gcm または chacha20poly1305
  - KDF: argon2（必要なら pbkdf2 を互換用に）
  - RNG: rand（OS乱数）
  - メモリ消去: zeroize
  - データ: serde

- マスターパスワードを誤ると解錠できない（改ざん/破損も検知できる）。

- CSVインポート/エクスポートが仕様通りに動作する。
- 検索が仕様通りに動作する（タグ、部分一致など）。
